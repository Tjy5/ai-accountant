# AI 自动分类功能改进说明

## 改进概述

本次改进主要解决了移动端 AI 自动分类功能总是将交易分类到"其他"的问题。通过引入智能模糊匹配算法和优化系统提示词，显著提升了 AI 分类的准确性。

## 主要改进内容

### 1. 添加智能模糊匹配功能

在 `backend/routes/ai.js` 中新增了 `findBestMatchingCategory` 函数，实现了多层次的分类匹配策略：

#### 匹配策略（按优先级）：

1. **精确匹配**（不区分大小写）
   - 例如：AI 返回 "餐饮" → 匹配用户分类 "餐饮"

2. **包含匹配**
   - AI 返回的分类包含在用户分类中
   - 例如：AI 返回 "餐" → 匹配用户分类 "餐饮"

3. **被包含匹配**
   - 用户分类包含在 AI 返回的分类中
   - 例如：AI 返回 "日常餐饮" → 匹配用户分类 "餐饮"

4. **描述匹配**
   - AI 返回的分类在用户分类的描述中
   - 例如：AI 返回 "打车" → 匹配用户分类 "交通"（描述：公共交通、打车等）

5. **编辑距离匹配**
   - 使用 Levenshtein 距离算法计算相似度
   - 编辑距离 ≤ 2 且相似度 > 60% 时匹配
   - 例如：AI 返回 "餐飲" → 匹配用户分类 "餐饮"

### 2. 优化系统提示词

改进了发送给 AI 的系统提示词，提供更详细的分类信息：

**改进前：**
```
category 必须从以下列表选择：餐饮, 交通, 购物, 工资, 其他
```

**改进后：**
```
category 必须从以下分类中选择最合适的：
   - 餐饮（日常餐饮消费）
   - 交通（公共交通、打车等）
   - 购物（日常用品购买）
   - 工资（工资收入）
   - 其他（其他未分类项目）
```

这样 AI 可以根据描述信息更准确地选择分类。

### 3. 优化分类数据获取

修改了 `getUserCategories` 函数，从数据库获取分类时同时获取描述和类型信息：

```javascript
// 改进前：只获取分类名称
SELECT name FROM categories WHERE user_id = ? AND deleted_at IS NULL

// 改进后：获取完整分类信息
SELECT name, description, type FROM categories WHERE user_id = ? AND deleted_at IS NULL
```

返回的数据结构从简单数组改为 Map 对象，包含完整的分类元数据。

## 技术实现细节

### Levenshtein 距离算法

实现了标准的 Levenshtein 距离算法，用于计算两个字符串之间的编辑距离：

```javascript
function levenshteinDistance(str1, str2) {
  const len1 = str1.length;
  const len2 = str2.length;
  const matrix = Array(len1 + 1).fill(null).map(() => Array(len2 + 1).fill(0));

  for (let i = 0; i <= len1; i++) matrix[i][0] = i;
  for (let j = 0; j <= len2; j++) matrix[0][j] = j;

  for (let i = 1; i <= len1; i++) {
    for (let j = 1; j <= len2; j++) {
      const cost = str1[i - 1] === str2[j - 1] ? 0 : 1;
      matrix[i][j] = Math.min(
        matrix[i - 1][j] + 1,
        matrix[i][j - 1] + 1,
        matrix[i - 1][j - 1] + cost
      );
    }
  }

  return matrix[len1][len2];
}
```

## 使用示例

### 示例 1：语音输入

**用户说：** "今天中午吃饭花了 50 块"

**AI 处理流程：**
1. AI 识别：type=expense, category="吃饭", amount=50
2. 模糊匹配：发现 "吃饭" 不在分类列表中
3. 智能匹配：通过描述匹配找到 "餐饮"（描述包含"餐饮消费"）
4. 最终结果：type=expense, category="餐饮", amount=50
5. 提示用户：分类 "吃饭" 已智能匹配为 "餐饮"

### 示例 2：拍照识别

**用户拍摄：** 滴滴出行的行程单

**AI 处理流程：**
1. AI 识别：type=expense, category="打车", amount=25.5
2. 模糊匹配：发现 "打车" 不在分类列表中
3. 智能匹配：通过描述匹配找到 "交通"（描述包含"打车"）
4. 最终结果：type=expense, category="交通", amount=25.5
5. 提示用户：分类 "打车" 已智能匹配为 "交通"

### 示例 3：相似匹配

**用户说：** "买了件衣服 200 元"

**AI 处理流程：**
1. AI 识别：type=expense, category="购买", amount=200
2. 模糊匹配：发现 "购买" 不在分类列表中
3. 智能匹配：通过编辑距离找到 "购物"（相似度高）
4. 最终结果：type=expense, category="购物", amount=200
5. 提示用户：分类 "购买" 已智能匹配为 "购物"

## 性能影响

- **数据库查询**：每次 AI 请求增加了 `description` 和 `type` 字段的查询，但通过缓存优化，实际影响很小
- **匹配算法**：模糊匹配算法的时间复杂度为 O(n*m)，其中 n 是分类数量，m 是字符串长度。对于典型的 10-20 个分类，性能影响可忽略不计
- **内存使用**：categoryMap 使用 Map 数据结构，内存占用略有增加，但在可接受范围内
